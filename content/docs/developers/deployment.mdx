The temetro backend doubles as the **wallet relay**: the patient's app connects to it (over the
`/wallet` Socket.io namespace) to complete a **scan-to-connect** import. For a phone to scan from
**any network** — not just the clinic's Wi-Fi — that backend needs a **public `https://` URL**.

This page covers deploying it. The backend ships a production **`Dockerfile`** that auto-generates
any missing secrets and runs database migrations on start, so every provider below just builds that
image.

<Callout type="info" title="Just testing?">
  You don't need to deploy to try off-network scanning. Run `npm run dev:tunnel` in `backend/` (needs
  `cloudflared`) — it opens a public tunnel to your local server and sets `PUBLIC_RELAY_URL` for you.
  See [Import from a patient app](/docs/guides/wallet-import).
</Callout>

## The environment contract

Whatever host you choose, set these:

| Variable | Value |
| --- | --- |
| `DATABASE_URL` | Postgres connection string (the provider's managed Postgres). |
| `PUBLIC_RELAY_URL` | The backend's **public `https://` URL** — this is baked into the wallet-import QR, so it must be reachable from the patient's phone. |
| `BETTER_AUTH_URL` | The same public `https://` URL (the `https` scheme switches auth cookies to **secure**). |
| `FRONTEND_URL` | Your deployed frontend origin (CORS + cookies + email links). |
| `BETTER_AUTH_SECRET`, `AI_CREDENTIALS_KEY` | Strong secrets. The Docker entrypoint generates them if unset, but persist them (a volume or provider-managed value) so sessions/keys survive redeploys. |

Repo files: `backend/fly.toml`, `backend/render.yaml`, `backend/railway.json`.

## Fly.io

```bash
cd backend
fly launch --no-deploy --copy-config        # keep fly.toml; pick app name + region
fly volumes create temetro_data --size 1    # persists secrets + uploads
fly postgres create
fly postgres attach <postgres-app-name>      # sets DATABASE_URL
fly secrets set \
  PUBLIC_RELAY_URL=https://<app>.fly.dev \
  BETTER_AUTH_URL=https://<app>.fly.dev \
  FRONTEND_URL=https://<your-frontend>
fly deploy
```

Fly's proxy upgrades WebSockets automatically; `fly.toml` keeps one machine always running so the
relay never sleeps mid-pairing.

## Render

`backend/render.yaml` is a Blueprint that provisions a managed Postgres and a Docker web service.

1. **New → Blueprint** and point Render at this repo. Because the backend is in a subdirectory, set
   the service **Root Directory** to `backend/` (or copy `render.yaml` to the repo root and prefix the
   paths with `backend/`).
2. Render injects `DATABASE_URL` and generates `BETTER_AUTH_SECRET` / `AI_CREDENTIALS_KEY`
   automatically.
3. After the first deploy, set `PUBLIC_RELAY_URL` and `BETTER_AUTH_URL` to
   `https://<service>.onrender.com` and `FRONTEND_URL` to your frontend origin, then redeploy.

Render supports WebSockets out of the box; the health check is `/health`.

## Railway

`backend/railway.json` builds the `Dockerfile` with a `/health` check.

1. **New Project → Deploy from GitHub repo**; set the service **Root Directory** to `backend/`.
2. Add a **Postgres** database to the project. Reference it from the backend service:
   `DATABASE_URL=${{Postgres.DATABASE_URL}}`.
3. Set `PUBLIC_RELAY_URL` and `BETTER_AUTH_URL` to your public Railway URL
   (`https://<service>.up.railway.app` or a custom domain) and `FRONTEND_URL` to your frontend
   origin. Generate a domain under the service's **Settings → Networking** if you haven't.

<Callout type="warn" title="PUBLIC_RELAY_URL must be the public https URL">
  This is the single value that makes scan-to-connect work off-network — it's what the QR tells the
  phone to dial. If it's a `localhost`/LAN address or `http://`, the phone can't connect (iOS also
  blocks cleartext `http`). When unset, the backend falls back to the request host and honors
  `x-forwarded-proto`, which works behind these proxies — but setting it explicitly is recommended.
</Callout>
